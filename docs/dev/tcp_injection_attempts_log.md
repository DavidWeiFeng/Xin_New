# 赛尔号代理：TCP 主动注入技术尝试日志

## 1. 项目背景
本项目旨在实现一个基于 `WinDivert` 的透明游戏代理。除了被动解析协议外，核心需求是实现**主动发送游戏指令**（如自动战斗），这要求我们在现有的 TCP 连接中凭空插入数据包，同时不破坏 TCP 协议栈的序列号（SEQ/ACK）连续性，不导致掉线。

## 2. 核心挑战
TCP 是一个有状态、可靠的流式协议。
*   **SEQ 同步**：插入数据会导致 SEQ 增加，若不通知客户端，客户端后续发包的 SEQ 会偏小，被服务器视为重传或乱序。
*   **ACK 确认**：服务器确认收到注入包后，会回复更大的 ACK。若该 ACK 传回客户端，客户端会发现 ACK > 自己已发的 SEQ，触发 RST 重置连接。
*   **校验与完整性**：CheckSum、Timestamp 等字段必须合法，否则会被静默丢弃。

## 3. 尝试方案记录

### 3.1 方案一：寄生替换 (Payload Replacement)
**思路**：
拦截客户端发出的下一个包（如心跳 CMD 1002），**丢弃**其原有内容，**替换**为注入指令（CMD 2408）。
**结论**：最稳健，但受限于心跳周期（约 10s），实时性差。

### 3.2 方案二：实时注入 (Active Injection)
**思路**：
在转发完客户端原包后，立即构造并发送一个新的 TCP 包。
**失败表现**：服务器不确认新数据，客户端随后卡死。
**根本原因**：未开启应用层序列号重写，导致注入包的 Result 字段与客户端后续包冲突；且对象复用可能导致底层 buffer 状态错乱。

### 3.3 方案三：双重分身发送 (Doppelgänger Sending) - **最终成功方案**
**思路**：
当检测到注入需求时，拦截当前的客户端原包（不发送），利用其作为载体生成两个“分身”：
1.  **分身 A (注入包)**：克隆原包头部，替换 Payload 为注入指令，SEQ 保持当前。
2.  **分身 B (修正后的原包)**：将原包的 SEQ 后移（加上分身 A 的长度），确保接续在注入包之后。
**关键突破点**：
*   **必须开启序列号重写**：确保整个会话的应用层 Sequence (Result) 完全由代理接管。
*   **对象克隆**：使用 `pydivert.Packet(packet.raw)` 创建独立的注入包对象，避免修改冲突。
*   **PSH 标志**：强制设置 TCP PSH 位，确保服务器立即处理。

## 4. 最新验证结果 (2026-01-31)
*   **注入过程**：成功捕捉到 Client -> Server 包，先后发出 `[Inject]` 和 `[Restore]`。
*   **连接稳定性**：服务器在收到注入序列后，**未断开连接**，并持续下发大量后续数据包（SEQ 持续增长）。
*   **SEQ/ACK 对齐**：服务器 ACK 能够跟随注入包的节奏移动（虽然观察到有 4 字节的微小偏移，但不足以破坏 TCP 会话）。
*   **结论**：**TCP 层面的注入难题已攻克**。现在的代理能够实现毫秒级的指令插入而不会导致游戏掉线。

## 5. 待优化与注意
1.  **4 字节 ACK 偏移**：初步怀疑与 WebSocket Mask Key 或 pydivert 内部包长计算有关，暂不影响稳定性。
2.  **业务触发逻辑**：确保 `monster_id` 等参数准确，以触发预期的服务器逻辑响应。
3.  **IP ID 管理**：当前已移除手动 ID 修改逻辑以规避属性名报错，若未来遇到防火墙拦截，可考虑通过 raw 字节方式微调 IP ID。
